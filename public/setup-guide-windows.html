§ Dev Setup Guide

Ship The Digest on Windows.

A step-by-step walkthrough for setting up your development pipeline from scratch — accounts, tools, local environment, and your first production deploy. No prior deployment experience needed.

Windows 10 or 11 ~45–60 minutes VS Code + Git already installed Git Bash terminal

Step 1.1
Create a GitHub account at github.com

Go to github.com → Sign up. Use your real email — GitHub will send you a verification link. Choose the free plan when asked.

Pick a professional username — it'll appear in your repo URLs and on your Vercel deployment links.
Step 1.2
Create a Vercel account at vercel.com

Go to vercel.com → Sign Up → Continue with GitHub. This connects the two accounts immediately — which you need anyway. Authorize Vercel when GitHub asks.

You're creating a Vercel account by logging in with GitHub. No separate password needed.
Step 1.3
Create an Anthropic account at console.anthropic.com

Go to console.anthropic.com → Sign up. You'll need to add a payment method to get an API key, but you won't be charged until you actually use it. You'll set a hard spending cap in a later step before running anything.

The Anthropic account for API access is separate from a Claude.ai subscription. You need both if you use Claude.ai — but only the API account is needed to run The Digest.

Step 2

Install Node.js.

Node.js is the runtime that lets you run JavaScript on your computer (not just in a browser). It also installs npm — the package manager you'll use to install project dependencies.

Step 2.1
Download and install Node.js LTS

Go to nodejs.org. You'll see two buttons — download the one labelled LTS (Long Term Support). This is the stable version. Run the installer and accept all defaults — you don't need to change anything.

LTS means it's battle-tested and won't break things unexpectedly. Avoid the "Current" version for production projects.
Step 2.2
Verify it installed correctly

Open VS Code. Open the terminal with Ctrl + ` (backtick — the key above Tab). Make sure it's using Git Bash (see next section if not). Run these two commands:

Git Bash
node --version
npm --version

You should see version numbers — something like v20.11.0 and 10.2.4. If you see "command not found", close and reopen VS Code, then try again — the installer sometimes needs a restart to take effect.

Step 3

Set up your terminal.

You already have Git Bash installed (it came with Git). This gives you Linux-style commands that every tutorial assumes — inside VS Code, so you never need to leave your editor.

Step 3.1
Make Git Bash the default terminal in VS Code

In VS Code: open the Command Palette with Ctrl + Shift + P. Type Terminal: Select Default Profile and press Enter. Select Git Bash from the list.

Now open a new terminal (Ctrl + `). You should see a prompt ending in $ — that's Git Bash. If it still shows PowerShell (the prompt ends in >), close the terminal and open a new one.

You can always check which shell you're in by looking at the dropdown in the top-right corner of the terminal panel.
Step 3.2
The five commands you'll actually use

You don't need to memorise the terminal. You need these five commands — everything else you can look up.

pwd
Show which folder you're currently in
ls
List files and folders here
cd folder-name
Move into a folder
cd ..
Go up one folder
Ctrl + C
Stop a running process (like a dev server)
Tab autocompletes folder and file names. Type the first few letters and press Tab — it fills in the rest.

Step 4

Fork & clone the repo.

Forking creates your own copy of the project on GitHub. Cloning downloads it to your computer. You'll work on your local copy, then push changes back to GitHub — which triggers an automatic deploy.

Step 4.1
Fork the repository on GitHub

Go to the repo at github.com/fde-shipit/ai-news-agent. In the top-right corner, click Fork → Create fork. GitHub creates a copy of the repo under your own account. You'll now be at github.com/YOUR-USERNAME/ai-news-agent.

Forking means you own your copy. You can change anything without affecting the original. You can also pull in future updates from the original whenever you want.
Step 4.2
Clone it to your computer

On your forked repo page, click the green Code button → make sure HTTPS is selected → click the copy icon next to the URL.

In VS Code's terminal, navigate to where you want to keep your projects (e.g. cd Documents) and run:

Git Bash
git clone https://github.com/YOUR-USERNAME/the-digest.git
cd the-digest

Git downloads the project and you move into it. You should now see all the project files in VS Code's sidebar.

Step 4.3
Configure Git with your identity

If you've never used Git on this machine, run these once — it tells Git who's making changes:

Git Bash
git config --global user.name "Your Name"
git config --global user.email "your@email.com"

Use the same email as your GitHub account.

Step 5 — Read this carefully

Configure your API key.

This is the most important section in the guide. API keys are passwords. How you store them determines whether your project stays secure or becomes expensive very quickly.

The golden rule: your API key must never be committed to Git. If it ends up on GitHub — even for a minute, even in a private repo — rotate it immediately at console.anthropic.com and generate a new one. Bots scan GitHub continuously for exposed keys.
Step 5.1
Get your Anthropic API key

Go to console.anthropic.com/settings/keys → Create Key. Give it a name (e.g. "the-digest-local"). Copy the key — it starts with sk-ant-. You can only see it once, so copy it now.

If you close the page before copying, just delete that key and create a new one. You can have multiple keys and delete them anytime.
Step 5.2
Create your .env.local file

In the project root (the the-digest folder), create a new file called .env.local. In VS Code you can right-click the file explorer sidebar → New File. Name it exactly .env.local — the dot at the start matters.

Add this content, replacing the placeholder with your real key:

.env.local
# Anthropic API key — never commit this file
ANTHROPIC_API_KEY=sk-ant-your-key-here

# Self-hosted mode (no login required)
NEXT_PUBLIC_MULTI_USER=false
Step 5.3
Verify .gitignore is protecting it

Open the .gitignore file in the project root. Look for a line that matches the pattern .env*.local — it should already be there. This tells Git to never track this file or other local env files, even if you run git add .

To double-check it's working, run:

Git Bash
git status

Your .env.local and other local env files should not appear in the output. If they do, the pattern is not in your .gitignore — add it before proceeding.

.env*.local vs .env — Next.js treats files matching .env*.local differently from plain .env files. The pattern .env*.local matches .env.local, .env.development.local, and other local env files. Always use this pattern for secrets on your machine.

Step 6

Run it locally.

Before deploying to the world, run it on your own machine first. This is where you test changes, catch errors, and see what users will see — without affecting anyone.

Step 6.1
Install project dependencies

In the terminal, make sure you're in the project folder (run pwd — you should see /c/Users/you/Documents/the-digest or similar). Then run:

Git Bash
npm install

This reads the package.json file and downloads all the libraries the project needs into a node_modules folder. It takes 1–2 minutes. You'll see a lot of text scrolling — that's normal.

node_modules can be hundreds of megabytes, but it's also in .gitignore — it never gets committed. Anyone who clones your repo just runs npm install themselves.
Step 6.2
Start the development server
Git Bash
npm run dev

Wait for the message ▲ Next.js ready on http://localhost:3000. Then open your browser and go to localhost:3000. You should see The Digest running.

To stop the server: click the terminal and press Ctrl + C.

The dev server reloads automatically when you save a file. You'll see changes in the browser instantly — you don't need to restart it.
Step 6.3
Test that it actually works end to end

Pick up to 2 topics, select a signal type, and click Generate. Watch the agent log. You should see web searches completing and a digest appearing. If you get an API error, check that your key in .env.local is correct and has no extra spaces.

Every successful generate call costs roughly $0.05–0.10. Run a few tests here, then set your spend limit in Step 8 before sharing it with anyone.

Step 6 (Optional)

Deploy with Claude Code.

If you have Claude Code installed in your editor, you can use it to walk through deployment interactively — no need to manually follow the steps.

Step 6 (Optional)
Run /deploy in Claude Code

Open the terminal in VS Code and run:

Git Bash
claude

Then type:

Claude Code will guide you through the entire deployment process interactively, asking clarifying questions and helping you set up your production environment step by step.

This is a hands-on guided experience. Claude Code can help troubleshoot issues, explain what's happening at each step, and catch mistakes before they cause problems.

Step 7

Deploy to Vercel.

Vercel watches your GitHub repo. Every time you push a change, it automatically rebuilds and redeploys your site. Your first deployment takes about 2 minutes.

Step 7.1
Import the project in Vercel

Go to vercel.com/new. You'll see a list of your GitHub repos. Find the-digest and click Import.

On the configuration screen: leave all settings as defaults. Next.js is detected automatically. Do not click Deploy yet — you need to add your environment variables first.

Step 7.2
Add your environment variables

On the same configuration screen, expand Environment Variables. Add each variable from your .env.local file — key on the left, value on the right:

Add these one at a time in Vercel's UI
ANTHROPIC_API_KEY         sk-ant-your-key-here
NEXT_PUBLIC_MULTI_USER    false
This is the safe way to store secrets in production. Vercel encrypts these values — they're never visible to anyone, including you, after you save them. This is very different from putting them in your code.
Step 7.3
Deploy

Click Deploy. Vercel builds your project — you can watch the logs in real time. When it finishes, you'll see a confetti animation and a live URL like the-digest-abc123.vercel.app.

Click the URL and test it. Generate a digest. Make sure it works exactly as it did locally.

You can add a custom domain later: in Vercel, go to your project → Settings → Domains. You can connect any domain you own.

Step 8 — Do this before sharing it

Set your spend limits.

Before you share the URL with anyone, cap your Anthropic spending. This is your backstop — if something goes wrong or traffic spikes unexpectedly, this is what stops a surprise bill.

Step 8.1
Set a monthly hard cap in Anthropic console

Go to console.anthropic.com/settings/billing → Usage limits. Set a monthly budget cap — $10–20 is sensible for a personal project. When this limit is hit, the API stops responding until the next billing period. Your site shows an error rather than charging you more.

At ~$0.07 per digest run, $20/month = roughly 280 runs. That's about 9 a day — plenty for personal use or a small audience.

Step 9 — The ongoing cycle

Making changes.

Once you're deployed, the cycle for every future change is the same four commands. Learn these and you own the whole pipeline.

Step 9.1
The edit → push → deploy cycle

Edit a file locally. Test it at localhost:3000. When you're happy, run:

Git Bash — run these in order
git add .
git commit -m "describe what you changed"
git push

That's it. Vercel detects the push and automatically deploys. Your live site updates in about 90 seconds. You can watch the progress at vercel.com/dashboard.

git add . stages all changed files. git commit -m "..." saves a snapshot with a message. git push sends it to GitHub. Vercel does the rest.
Step 9.2
Adding or changing environment variables

Local changes: edit your .env.local file, then restart the dev server (Ctrl + C, then npm run dev).

Production changes: go to your project in the Vercel dashboard → Settings → Environment Variables. Edit the value, save, then go to Deployments and click Redeploy on the latest deployment to pick up the new value.

Vercel doesn't automatically pick up env var changes — you need to trigger a redeploy after changing them.

Reference

Common errors.

If something breaks, it's almost always one of these.

Error
"command not found: npm" or "command not found: node"

Node wasn't added to your PATH, or VS Code didn't pick up the new PATH yet. Close VS Code completely and reopen it. If that doesn't fix it, restart your computer, then try again.

Error
401 Unauthorized from the API

Your API key is wrong or missing. Check .env.local — make sure the key starts with sk-ant- and has no extra spaces or line breaks. After editing .env.local, restart the dev server.

Error
Works locally but fails on Vercel

Almost always missing environment variables in Vercel. Go to your project → Settings → Environment Variables. Check that ANTHROPIC_API_KEY is there and spelled exactly right (case-sensitive). Then redeploy.

Error
Vercel build fails with a red error

Click on the failed deployment in Vercel → expand the build logs → look for the first red line. Copy that error into Claude and ask what it means. Build errors are almost always specific and fixable quickly.

The Vercel build log is one of the most useful debugging tools you have. Don't skip past the red lines — the first one is usually the actual problem.
Error
"Permission denied" when running Git commands

GitHub requires authentication. Run git push — it'll open a browser window asking you to sign in to GitHub. Authorise it. After that first sign-in, pushes work silently. If you're on a machine that's already signed in to GitHub via VS Code, this usually sorts itself automatically.
